[[section-technical-risks]]
== Risks and Technical Debts

=== Risks

[options="header",cols="1,1,3,3"]
|===
|ID |Priority |Risk |Mitigation

|RISK_1
|High
|**Pixlr Vendor Lock-in:** The app relies on Pixlr for its core editing functionality (<<REQ_7>>). If Pixlr changes its API, pricing, or discontinues the service, the editing feature is at risk.
|Adapter Pattern (see ADR-3) isolates Pixlr dependency. Maintain a list of alternative providers (e.g. Cloudinary, imgix). Monitor Pixlr API changelog.

|RISK_2
|High
|**Image Data Scaling:** As the user base grows, image storage and bandwidth costs can escalate rapidly. A single viral post could cause unexpected load spikes.
|Use cloud object storage with auto-scaling and CDN. Implement image compression and thumbnail generation. Set up cost alerts and usage monitoring.

|RISK_3
|High
|**GDPR Compliance:** EU users' personal data must be handled according to GDPR. Non-compliance can result in significant fines.
|Implement data deletion endpoints, consent management, and data export functionality. Store EU user data in EU regions. Regular compliance audits.

|RISK_4
|Medium
|**React Native Performance for Image Processing:** Heavy image manipulation operations may cause UI lag on lower-end devices.
|Offload intensive processing to backend services. Use native modules for critical image operations. Implement lazy loading and pagination.

|RISK_5
|Medium
|**Microservice Operational Complexity:** With a small team, managing multiple services, deployments, and inter-service communication adds overhead.
|Use container orchestration (Docker Compose / Kubernetes). Implement centralized logging and monitoring. Start with fewer services and split only when necessary.
|===

=== Technical Debt

[options="header",cols="1,1,3,3"]
|===
|ID |Priority |Debt |Planned Resolution

|TD_1
|Medium
|**No automated end-to-end testing:** Current focus is on unit tests. Integration and E2E tests are not yet in place.
|Introduce Detox (React Native E2E) and API integration tests before first production release.

|TD_2
|Low
|**Monolithic database schema:** All services currently share a single PostgreSQL database, contradicting the microservice approach.
|Plan database-per-service migration once team capacity allows. Start with logical schema separation.
|===
