== Architecture Decisions

=== ADR-1: Microservice Architecture

* **Status:** Accepted
* **Context:** The Image Sharing App has distinct functional domains (user management, image handling, community features, editing) with different scaling requirements. The start-up needs to iterate quickly on individual features.
* **Decision:** Decompose the system into four microservices: User Service, Image Service, Community Service, and Editing Service.
* **Consequences:**
** [+] Independent deployment and scaling of each service
** [+] Technology flexibility per service
** [-] Increased operational complexity (service discovery, inter-service communication)
** [-] Distributed data management requires careful consistency handling

=== ADR-2: React Native for Cross-Platform Mobile Development

* **Status:** Accepted
* **Context:** The app must be available on both iOS and Android. The team is small (3 developers) and cannot maintain two native codebases.
* **Decision:** Use React Native as the mobile framework.
* **Consequences:**
** [+] Single codebase for both platforms, reducing development time by ~40%
** [+] Large community and ecosystem
** [-] Slight performance trade-off compared to fully native apps for image-heavy operations
** [-] Dependency on React Native release cycle for platform-specific features

=== ADR-3: Pixlr Integration via Adapter Pattern

* **Status:** Accepted
* **Context:** The app requires cloud-based photo editing capabilities (<<REQ_7>>). Pixlr offers a comprehensive REST API for this. However, tight coupling to a third-party API creates vendor lock-in risk.
* **Decision:** Integrate Pixlr through an Adapter Pattern in the Editing Service, defining an internal editing interface that Pixlr implements.
* **Consequences:**
** [+] Loose coupling: Pixlr can be replaced with an alternative provider by implementing a new adapter
** [+] Internal API remains stable regardless of Pixlr API changes
** [-] Additional abstraction layer adds development effort
** [-] Not all Pixlr-specific features may be easily abstracted

=== ADR-4: OAuth 2.0 with Social Login

* **Status:** Accepted
* **Context:** The target audience (18–35, millennials/Gen Z) expects frictionless registration. Security must comply with industry standards and GDPR.
* **Decision:** Use OAuth 2.0 for authentication with Google and Facebook social login options, plus traditional email/password.
* **Consequences:**
** [+] Reduced registration friction increases conversion rates
** [+] Delegated authentication reduces security responsibility for credential storage
** [-] Dependency on third-party identity providers (Google, Facebook)

=== ADR-5: Cloud Object Storage for Images

* **Status:** Accepted
* **Context:** The app handles large volumes of image data that need to be stored cost-effectively and delivered globally with low latency.
* **Decision:** Store all image binaries in cloud object storage (AWS S3 / GCP Cloud Storage) with CDN distribution.
* **Consequences:**
** [+] Virtually unlimited, cost-effective storage
** [+] Built-in redundancy and high availability
** [+] CDN integration for fast global delivery
** [-] Egress costs can become significant at scale
